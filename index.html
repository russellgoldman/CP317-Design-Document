Definitions

Users

Person that uses the BeerBuddy application

Experienced Drinkers

Users that have a general understanding of what they want in an alcoholic beer beverage and need more validation on beers that suit their preferences

Novice Drinkers

Users that don’t know what aspects they prefer in a beer and need a reference to popular choices

abv

Alcohol percentage by volume



1. Introduction


1.1 The Purpose of the Document

The purpose of this document is to make a comprehensive design of how the BeerBuddy app will work on both the front end and the back end. This will draw an approximate picture of the connections among both ends and interfaces.


1.2 The Scope of the Document

This document is aimed at developers involved in the development of the application.


1.3 Overview of the Application

Users will be greeted by a screen giving them an option to select their beer drinking experience being either novice (not knowing their preferences in beer) or experienced (knowing their preference in beer). Experienced drinkers will be redirected to filter options such as beer colour, alcohol percentage (by volume), country of origin, and availability in can/bottle/keg. They will then receive a list of beers that match their filters ranging from most-alike to least-alike. Novice drinkers will be redirected to a page of top picks from beer types such as lagers, malts, stouts, and ales. Top picks will be referenced from the backend database. Users will also be allowed to add beer by filling out the beverages characteristics on a separate page.

2. Core Structures


2.1 Application Programming Interface (API)

Front-end and back-end are separated. Interaction with the application will only be done through front-end calls to the back-end Application Programming Interface (API). The front-end is composed of two pages including an introduction page to allow the user to select the novice or experienced drinkers edition of the BeerBuddy application. The novice edition will offer less advanced features more suited to discover a novice users preferences, whereas the experience version will use more beer associated vocabulary and offer more versatility in the user’s experience. Backend API of the application will coordinate data transfer between the client and the server.


2.2 API Specifications

The API will be written using the Flask framework, which will connect the database and the front-end Django client.

4. User Interface Design


4.1 Overview

The user interface consists of the landing page, filters page, resulted query page, individual beer viewing pages with beer ratings, and finally the add beers page.


4.2 Navigation

The navigation of the page is achieved by providing a responsive navigation bar at the top of every page. The navigation bar is based off components provided by bootstrap.


4.3 The Landing Page (HomeView)

The page will have a navigation bar and two buttons to choose from. Novice user or experienced users. Users will have a chance to read about us and contact us. They will also have a chance to login if they had an existing account. This will be achieved through the navigation bar at the top of every page.



# Field Type Description
1 Novice Button Links to the main page for novice users
2 Experienced Button Links to the main page for experienced users


4.4 The Filters Page (Experienced - SearchBeerView)

The page lets the users search for beer by selecting the alcohol percentage by volume, brand, body type, container type and taste of the beer. It will generate all records of beers in our database that match the search criteria. The result of the beer is displayed in the results page.



# Field Type Description

2 Submit Button Lets the user select the type of beer
3 Brand Dropdown box Lets the user select the brand of the beer
4 ContainerType Multiselect box Lets the user select the ContainerType(s) of the beer.
5 Taste Multiselect box Lets the user select the Taste(s) of the beer


4.5 The Library page (Experienced - BeerResultsView)

User can change their choice options and re-do search (if user chooses nothing, all beers will be displayed).





4.6 The add beer page (All users - CreateBeerView)

The page will have a form to input information about the beer the user wants to submit.

5. Functionality


5.1 Search

  • User will be able to search for beers through filters such as color and brand
  • User can search through keywords which will match through description and stringified categories
  • If user select nothing, nothing will be displayed

5.2 Display

  • User will be able to see a webpage for each beer which contains information about the beer such as color and brand, which is displayed like a list.

5.3 Add/Edit Beer

  • User will be able to add beers to the database through an undetermined system as of now
  • User will be able to edit the description and categories of existing beers

6.5 Back-end Route Dictionary (Currently Implemented)


Name + Method Type Parameters + Return Type Description
Name:
GET /filteredBeer

Type:
API Route 		Parameters:
userDefinedFilters (Dictionary Object)

Return Type:
Dictionary Object 		The frontend sends a POST request to the backend with the user defined filters as a dictionary object.

The backend then uses the given dictionary object containing the filters to compare against all beers in the database and create a new dictionary object of all beers and their accuracies in relation to the filters.

Finally, the backend returns the new dictionary object if all worked appropriately, otherwise it returns an empty dictionary object with a 400 status code (BAD_REQUEST).

6.6 Back-end Route Dictionary (Future Implementations)


Name + Method Type Parameters + Return Type Description
Name:
POST /anonymousRating

Type:
API Route 		Parameters:
anonUserName (String), beerName (String), message String), rating (Integer; 0-5)

Return Type:
Void 		The frontend sends a POST request to the backend containing a dictionary object with the anonymous user's name (String), the beer name (String), the rating message (String), and the rating from 0-5 (Integer) as a parameter.

The backend receives the dictionary object, checks that see that the comment is not a complete duplicate, and then either writes the data to the database or does not.
Name:
GET /beerRatings

Type:
API Route 		Parameters:
beerName (String)

Return Type:
Dictionary Object 		The frontend sends a GET request to the backend to fetch the ratings for the currently viewed beer based on its name.

The backend receives the request and queries a table in the database containing all the ratings for all beers and returns all beer rating objects for the beer in question in an array of beer rating dictionary objects.

If the beer ratings are found, a populated dictionary object containing beer ratings is received, otherwise an empty dictionary object.
